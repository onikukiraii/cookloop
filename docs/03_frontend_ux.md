Core.md の「3つの主要体験」をフロントエンドでどう実現するかを定義する。

| 画面 | パス | 対応する主要体験 | 位置づけ |
|---|---|---|---|
| 冷蔵庫 | `/` | 冷蔵庫を把握する | メイン画面・ホーム |

## 1. 冷蔵庫（FridgePage）

### 目的

「冷蔵庫を開けなくても中身がわかる」を実現する画面。
解決する摩擦：**何がいつまでに使わないといけないかわからない**

### ユーザーの主な利用シーン

- 朝や夕方に「今日何使わなきゃいけないっけ」と確認する
- 料理後に残量を更新する（たっぷり → 半分 → 少し）
- 使い切った食材を削除する

### 表示の優先度

- **期限が近い食材を最も目立たせる**。左ボーダーの色で鮮度を視覚化（safe/warning/danger/expired）
- ソート順は賞味期限の昇順（期限が近い順）が自然
- 残量バッジは一目で「まだあるか、もうないか」を判断できる粒度

### 操作フロー

```
一覧を見る
  → 期限が近い食材に気づく
  → 「これ今日使おう」→ 献立提案へ遷移（※未実装）
  → 料理後に残量をタップして更新（full → half → little のサイクル）
  → 使い切ったらゴミ箱アイコンで削除
```

### 食材追加フロー

食材の追加は IngredientSelect コンポーネントで行う。

- **現状**：食材マスタに登録済みの食材から選択する方式
- **should**：選択時に OpenSearch でインクリメンタルサーチできるようにする。食材数が増えたときにリストから探すのは現実的でない
- **should**：マスタ未登録の食材名を入力した場合、Gemini で賞味期限を推定してマスタに自動登録し、そのまま冷蔵庫に追加できるシームレスなフローにする（現状は「食材マスタに先に登録 → 冷蔵庫で選択」の二度手間が発生する）

### 画面間連携

| 連携先 | トリガー | 動作 |
|---|---|---|
| 買い物リスト → 冷蔵庫 | 買い物リストでチェック | 冷蔵庫に自動登録される |
| 冷蔵庫 → 買い物リスト | 残量が「少し」になった定番食材 | 買い物リストに自動追加（※未実装） |
| 冷蔵庫 → 献立提案 | 「この食材を使いたい」 | 食材を選択した状態で献立提案へ遷移（※未実装） |

### 現状の実装

- CRUD は完成（一覧・追加・残量トグル・削除）
- 鮮度バッジ・残量バッジは実装済み
- 追加時は IngredientSelect で食材マスタから選択、賞味期限はオプション入力

### 未実装・検討が必要な体験

- [ ] **食材追加時の検索（should）**：IngredientSelect に OpenSearch インクリメンタルサーチを導入。入力文字列で食材名・aliases を検索し、候補を表示する
- [ ] **マスタ未登録食材の即時追加（should）**：検索してヒットしない場合、「新しい食材として追加」ボタンを表示 → Gemini で賞味期限推定 → マスタ登録 → 冷蔵庫に追加、の一連フロー
- [ ] **残量「少し」→ 買い物リスト自動追加の連携**：残量トグルで `little` にしたとき、定番フラグありの食材を自動で買い物リストに追加する。UI上のフィードバック（toast で「買い物リストに追加しました」）が必要
- [ ] **食材カードから献立提案への導線**：「この食材を使い切りたい」→ 献立提案画面に食材指定で遷移できるリンクやボタン
- [ ] **ソート順の明示**：現状APIの返却順に依存。期限昇順であることをUIで示すか、ソート切替を提供するか
- [ ] **グルーピング表示**：「今日・明日が期限」「2日以内」「それ以降」のようにセクション分けすると、緊急度がより伝わる
- [ ] **同一食材の複数登録**：同じ食材を2回買った場合の表示。現状は別カードとして表示される（意図通りか要確認）

## 2. 買い物リスト（ShoppingPage）

### 目的

スーパーで「何を買えばいいか」を即座に確認できる画面。
解決する摩擦：**「あれ、家に何があったっけ」と悩む時間**

### ユーザーの主な利用シーン

- スーパーに着いたらこの画面を開く
- 買ったものをチェックしていく
- 帰宅後、チェック済みアイテムは冷蔵庫に自動登録されている

### 表示の優先度

- **未購入アイテムのみ表示する**（購入済みアイテムは非表示）
- Core.md の「2セクション構成」：【買うもの】と【そろそろなくなりそう】に分ける
- 各アイテムには source バッジを表示し、リストに入った理由を明示する：
  - **手動**：ユーザーが自分で追加したもの
  - **レシピ**：献立提案から「この料理に必要」として追加されたもの（※未実装）
  - **定番自動**：定番食材の残量が「少し」になり自動追加されたもの（※未実装）

### 操作フロー

```
リストを確認（未購入のみ表示）
  → スーパーで商品を手に取ったらチェック
  → チェックした瞬間に冷蔵庫へ自動登録（toast で通知）
  → 買わないことにしたアイテムは削除
```

### 画面間連携

| 連携先 | トリガー | 動作 |
|---|---|---|
| 買い物リスト → 冷蔵庫 | チェック完了 | 冷蔵庫に自動登録（賞味期限はマスタのデフォルト値） |
| 冷蔵庫 → 買い物リスト | 定番食材が「少し」に | staple_auto として自動追加 |
| 献立提案 → 買い物リスト | 不足食材の追加 | recipe ソースとして追加（※未実装） |

### 現状の実装

- 未購入 / 購入済みの2セクション表示は実装済み
- チェック → 冷蔵庫自動登録の連携は実装済み
- source バッジ（手動 / レシピ / 定番自動）の表示は実装済み

### 未実装・検討が必要な体験

- [ ] **購入済みアイテムの非表示化**：現状は購入済みセクションとして表示されている。購入済みは非表示にし、一覧は未購入のみとする
- [ ] **2セクション構成（Core.md 準拠）**：【買うもの（manual + recipe）】と【そろそろなくなりそう（staple_auto）】にセクション分け
- [ ] **献立提案からの不足食材追加**：`POST /recipe/suggest/add-shopping` の UI。提案画面で「この料理を作る → 足りない食材をリストに追加」のフロー

## 3. 献立提案（MenuSuggestPage）

### 目的

「今夜何作ろう」の意思決定を AI に委ねる画面。
解決する摩擦：**毎日の献立をゼロから考える疲れ**

### ユーザーの主な利用シーン

- 夕方「今夜何作ろう」と思ったときにこの画面を開く
- 冷蔵庫の食材一覧を見て「提案してもらう」ボタンを押す
- 3つの候補から1つ選ぶ → 手順を見て料理する

### 表示の優先度

- **提案結果が主役**。冷蔵庫の食材一覧はコンテキストとして小さく表示
- ホットクックレシピにはメニュー番号を大きく表示（ホットクックに入力するため）
- 各提案に「使う食材」「省略・代替の注記」「手順」を添える

### 操作フロー

```
画面を開く → 冷蔵庫の食材が自動表示
  → パターンA：「今夜何作ろう」→ ボタンを押す → 3候補が表示
  → パターンB：「この食材を使いたい」→ 食材を選択 → ボタンを押す → 3候補
  → 候補を選ぶ → 手順が展開表示される
  → ホットクック → メニュー番号を確認して調理
  → 手動調理 → AI 生成の手順を見ながら調理
  → （オプション）足りない食材があれば買い物リストに追加
```

### 提案UIの切り替え

「おまかせ」と「食材指定」の2パターンは**タブで切り替える**。

### 画面間連携

| 連携先 | トリガー | 動作 |
|---|---|---|
| 冷蔵庫 → 献立提案 | 画面表示時 | 冷蔵庫の食材を自動取得して表示 |
| 冷蔵庫 → 献立提案 | 食材カードの「使い切りたい」 | 指定食材が選択された状態で遷移（※未実装） |
| 献立提案 → 買い物リスト | 「不足食材をリストに追加」 | recipe ソースで買い物リストに追加 |

### 現状の実装

- 冷蔵庫の食材一覧表示のみ。**AI 提案機能は未実装**（「準備中」表示）

### 未実装・検討が必要な体験

- [ ] **AI 提案の基本フロー**：`POST /recipe/suggest` の呼び出しと結果表示
- [ ] **タブによる提案モード切り替え**：「おまかせ」タブと「食材指定」タブ
- [ ] **提案結果カードのデザイン**：type（hotcook / manual）の視覚的区別、メニュー番号の目立たせ方、使用食材のハイライト
- [ ] **手順の展開表示**：カードクリックで手順をアコーディオン表示。hotcook は DB の手順、manual は AI 生成の手順
- [ ] **不足食材の買い物リスト追加**：提案結果に「冷蔵庫にない食材」が含まれる場合、ワンタップで買い物リストに追加
- [ ] **ローディング体験**：AI 応答は数秒かかるため、スケルトンやストリーミング表示の検討
- [ ] **提案の再生成**：結果が気に入らなければ「別の候補を見る」で再提案

## 4. 調味料（CondimentPage）

### 目的

食材とは別軸で調味料の在庫を管理する画面。
解決する摩擦：**「醤油切れてた」に料理中に気づく**

### ユーザーの主な利用シーン

- 調味料が減ったときに残量を更新
- たまに新しい調味料を追加
- 頻度は低い（週1回程度の確認で十分）

### 表示の優先度

- 残量が「少し」のものを上部に表示し、「切れそうなもの」を目立たせる
- 定番バッジはほぼ全品目に付くため、むしろ「定番でないもの」が例外的に目立つ形
- レシピ提案には使わないため、献立提案画面への導線は不要

### 操作フロー

```
一覧を見る
  → 残量をタップして更新（full → half → little のサイクル）
  → 残量「少し」の定番調味料は上部にソートされ目立つ → 買い物前に確認
```

### 画面間連携

| 連携先 | トリガー | 動作 |
|---|---|---|
| （買い物リスト連携なし） | — | 調味料は買い物リストに含めない。調味料画面で残量「少し」を上部ソートし、買い物前に確認する運用 |

### 現状の実装

- CRUD 完成（一覧・追加・残量トグル・削除）
- 定番バッジ表示済み

### 未実装・検討が必要な体験

- [ ] **買い物リストとの連携は不要**：調味料は買い物リストには含めない方針。代わりに残量「少し」のものを上部にソートし、買い物前にこの画面を確認する運用とする
- [ ] **残量「少し」のものを上部にソート**：切れそうな調味料を目立たせる
- [ ] **利用頻度が低い画面の位置づけ**：サイドバーの下部やサブメニューに置くなど、メイン導線から外すことも検討

## 5. 食材マスタ（IngredientMasterPage）

### 目的

食材の知識ベースを管理する設定画面。
ユーザーが直接使う頻度は低い（マスタはスクレイピング・Gemini で自動的に育つ。加えて、冷蔵庫追加時にマスタ未登録の食材を直接追加できるフローを整備することで、この画面を意識せずに済むようにする）。

### ユーザーの主な利用シーン

- Gemini が推定した賞味期限がおかしいときの修正
- 定番フラグの切り替え
- aliases（表記揺れ）の確認・修正

### 表示の優先度

- テーブル形式で一覧性を重視（現状の実装が適切）
- 検索・フィルタがあると便利（食材数が増えるため）

### 操作フロー

```
一覧を見る（ほぼ確認目的）
  → 必要に応じて賞味期限や定番フラグを編集
  → 定番フラグはワンタップで切り替え可能にする（should）
  → 新しい食材を手動追加（稀）
```

### 現状の実装

- テーブル形式の一覧・追加は実装済み
- 編集・削除は未実装

### 未実装・検討が必要な体験

- [ ] **定番フラグのワンタップ切り替え（should）**：テーブル上で定番フラグをワンタップで ON/OFF できるようにする。常備したい食材をすぐにマークできる体験
- [ ] **インライン編集**：賞味期限の日数をテーブル上で直接編集
- [ ] **aliases の表示・編集**：OpenSearch に保存された別名の確認 UI。誤った別名の修正（Core.md で言及あり）
- [ ] **検索・フィルタ**：食材数が数百になるため、テーブルの検索機能

## 画面横断の体験課題

### 食材ライフサイクルの一貫性

Core.md が描くライフサイクル全体がフロントで繋がっているか。

```
買い物リスト追加 → スーパーでチェック → 冷蔵庫登録 → 日々使う → 少し → 買い物リスト
       ↑                                                              ↓
       └──────────────────── ループ ──────────────────────────────────┘
```

**現状繋がっている部分：**
- チェック → 冷蔵庫登録（実装済み）

**まだ切れている部分：**
- 残量「少し」→ 買い物リスト自動追加（食材のみ。調味料は調味料画面で確認する運用）
- 献立提案 → 不足食材 → 買い物リスト追加
- 冷蔵庫 → 献立提案への食材指定遷移

### 新規食材の登録フロー

現状は「食材マスタに先に登録 → 冷蔵庫や買い物リストで選択」という二度手間が発生する。
理想は「冷蔵庫追加時に食材名を入力 → OpenSearch で検索 → ヒットしなければ Gemini 推定 → マスタ自動登録 → 冷蔵庫に追加」のシームレスなフロー。

### スマートフォンでの利用

スーパーで使う買い物リストは特にモバイル最適化が重要。
片手でチェックしやすい UI（チェックボックスのタップ領域、スワイプ操作など）。

### 通知・プッシュ

「明日期限の食材があります」のような通知は、アプリの価値を高める可能性があるが、個人利用 × ローカル環境という制約の中でどこまでやるか。
